# Bug修复流程

该skill定义了一个严格的7步bug修复流程。必须按照以下步骤顺序执行，每步都需要用户确认后才能进入下一步。

## 流程概述

```
步骤1: 分析代码 → 用户确认
    ↓
步骤2: 设计测试方案 → 用户确认
    ↓
步骤3: 确保现有UT通过
    ↓
步骤4: 添加失败UT重现bug
    ↓
步骤5: 修复代码（最小化修改）
    ↓
步骤6: 使新增UT通过
    ↓
步骤7: 使所有UT通过
```

---

## 步骤1: 分析代码并撰写说明

### 操作要求
1. **定位代码**: 根据bug描述，搜索并找到相关代码文件
2. **理解逻辑**: 仔细阅读代码，理解当前实现逻辑
3. **分析原因**: 找出导致bug的根本原因
4. **撰写说明**: 撰写一份详细的说明文档，包含：
   - 问题描述（bug的表现）
   - 涉及文件和代码位置
   - 当前代码逻辑分析
   - 根因分析（为什么会产生这个bug）
   - 建议的修复方向（不要给出具体代码，只给方向）

### 说明文档格式
```markdown
## Bug分析说明

### 问题描述
[bug的表象和症状]

### 涉及代码
- 文件: [文件路径]
- 方法/类: [方法名或类名]
- 行号范围: [起始行-结束行]

### 当前逻辑分析
[代码当前如何工作]

### 根因分析
[为什么会出现这个bug]

### 建议修复方向
[大致的修复思路，不涉及具体代码]
```

### 用户确认
将说明文档展示给用户，等待用户确认。用户确认后才可进入步骤2。

**可能的用户反馈处理**:
- 如果用户指出分析有误 → 重新分析并修正说明
- 如果用户要求补充信息 → 补充后再等待确认
- 如果用户确认无误 → 进入步骤2

---

## 步骤2: 设计测试用例方案

### 操作要求
1. **基于分析**: 根据步骤1的分析结果设计测试方案
2. **测试目标**: 设计的测试用例必须能重现该bug（即测试会失败）
3. **覆盖场景**: 考虑正常情况和边界情况
4. **撰写方案**: 不直接写UT代码，而是写测试用例方案文档

### 测试用例方案格式
```markdown
## 测试用例方案

### 测试目标
验证 [具体bug场景] 已被修复

### 测试用例1: [描述]
- **输入**: [具体输入值/状态]
- **预期输出**: [期望的正确输出]
- **当前行为**: [bug导致的错误输出]
- **验证点**: [测试要断言的内容]

### 测试用例2: [边界情况描述]
- **输入**: [边界条件]
- **预期输出**: [期望输出]
- **验证点**: [断言内容]

### 测试用例3: [异常情况描述，如适用]
...
```

### 用户确认
将测试用例方案展示给用户，等待用户确认。用户确认后才可进入步骤3。

**注意**: 此时不编写实际的UT代码，只提供方案。

---

## 步骤3: 确保现有单元测试通过

### 操作要求
1. **识别测试命令**: 确定项目使用的测试框架和运行命令
   - Maven: `mvn test`
   - Gradle: `./gradlew test`
   - Jest: `npm test`
   - pytest: `pytest`
   - 等等

2. **运行测试**: 执行完整的单元测试套件
3. **检查状态**: 确认所有现有测试是否通过

### 处理不同情况

**情况A: 所有测试通过**
- 告知用户当前测试状态
- 进入步骤4

**情况B: 有测试失败但与当前bug无关**
- 分析失败是否由当前bug引起
- 如果不是，向用户报告这些失败
- 询问用户是否继续（可能环境有问题）

**情况C: 有测试失败且与当前bug相关**
- 向用户说明这些失败
- 如果用户确认继续 → 进入步骤4

### 用户确认
向用户报告测试结果，等待用户确认进入下一步。

---

## 步骤4: 添加失败的单元测试

### 操作要求
1. **编写测试**: 根据步骤2确认的测试用例方案，编写单元测试代码
2. **确保失败**: 编写的测试必须能够失败，从而重现bug
3. **测试位置**: 遵循项目惯例（如放在对应测试类中）
4. **注释说明**: 在测试方法上添加清晰的注释，说明这是用于重现哪个bug

### 测试代码规范
```java
// 示例（Java/JUnit）
/**
 * Bug修复测试: [简述bug]
 * 对应步骤2的测试用例方案
 */
@Test
public void testBugScenarioX() {
    // Given: 设置测试数据
    [输入数据准备]

    // When: 执行被测方法
    [调用被测代码]

    // Then: 验证期望结果（此时会因bug而失败）
    [断言验证]
}
```

### 验证测试失败
- 运行新增的测试，确认它确实失败
- 如果测试通过 → 说明bug可能已被修复，或测试设计有误，需要向用户说明

### 用户确认
向用户展示新增的测试代码，报告测试结果（确认测试失败），等待用户确认进入下一步。

---

## 步骤5: 修复代码

### 操作要求
1. **最小化修改**: 仅修改必要的代码来修复bug，不添加额外逻辑
2. **全面考虑**: 修复时要考虑：
   - 当前方法的其他调用场景
   - 相关类的其他方法
   - 可能的副作用
3. **代码质量**: 保持代码风格一致性
4. **不要重构**: 专注于修复bug，不做无关的重构

### 修复原则
- **单一职责**: 一个修复只解决一个问题
- **向后兼容**: 不改变现有API的契约（除非必要）
- **防御性编程**: 考虑边界情况和异常输入

### 修改前确认
如果修复涉及较大改动或可能带来副作用，先向用户说明修改方案，等待确认。

### 修改后报告
向用户报告：
- 修改了哪些文件
- 修改了哪些行
- 修复的核心逻辑
- 潜在的风险点（如果有）

---

## 步骤6: 使新增单元测试通过

### 操作要求
1. **运行新增测试**: 仅运行步骤4添加的新测试
2. **验证通过**: 确认测试通过
3. **失败处理**: 如果测试仍失败
   - 分析失败原因
   - 检查修复是否完整
   - 可能需要返回步骤5

### 报告结果
向用户报告新增测试的运行结果，确认测试通过后进入最终步骤。

---

## 步骤7: 使所有单元测试通过

### 操作要求
1. **运行全部测试**: 执行完整的单元测试套件
2. **验证通过**: 确认所有测试（包括新增的）都通过
3. **失败处理**:
   - 如果新增的测试破坏其他测试 → 返回步骤5调整修复
   - 如果有其他测试失败 → 分析并修复

### 最终报告
向用户报告最终测试结果，确认：
- 所有现有测试通过
- 新增测试通过（重现并验证了bug修复）
- 没有引入回归问题

---

## 注意事项

### 每步必须用户确认
- 除非用户明确说"继续"、"下一步"或"确认"，否则不要自动进入下一步
- 每个步骤完成后，明确询问用户是否继续

### 失败回退
如果在某一步遇到问题，可以回退到上一步：
- 步骤4发现测试无法失败 → 返回步骤2调整测试方案
- 步骤5修复后测试仍失败 → 重新分析
- 步骤7发现回归问题 → 返回步骤5调整

### 代码修改范围
严格遵循最小修改原则：
- 修复bug本身
- 不添加未要求的功能
- 不做代码风格重构
- 不修改无关代码

### 测试优先
始终坚持"先写失败测试，再修复代码"的原则，确保：
1. 测试确实能捕捉到bug
2. 修复确实能解决问题
3. 测试可以作为回归保护

---

## 用户指令处理

当用户说以下话时，对应处理：

| 用户指令 | 处理 |
|---------|------|
| "修复bug" / "修bug" | 询问bug描述或相关代码位置，开始步骤1 |
| "继续" / "下一步" / "确认" | 进入当前步骤的下一步 |
| "返回上一步" / "回退" | 回到上一步重新开始 |
| "跳过步骤X" | 向用户说明风险，如果用户坚持则跳过 |
| "直接修复" | 坚持按照流程执行，向用户解释测试优先的重要性 |
